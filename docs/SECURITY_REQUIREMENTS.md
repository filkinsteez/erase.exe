# Security Requirements

## Security posture

This app performs destructive actions on a connected account. Security and user comprehension are product features.

## OAuth and account access

- Use OAuth 2.0 Authorization Code with PKCE.
- Never ask for or store an X password.
- Request the minimum scopes needed.
- Keep access tokens and refresh tokens server-side only.
- Encrypt tokens at rest with a managed key or a 32 byte AES-GCM key.
- Support token revocation and local data purge.
- Bind stored tokens to the authenticated app user and X user ID.

## Browser safety

- Never expose access tokens to client JavaScript.
- Never allow client-selected post IDs to be deleted without server-side ownership validation.
- Use CSRF protection for destructive job creation.
- Use secure, HTTP-only cookies for temporary OAuth state and PKCE verifier.
- Use same-site cookies where possible.

## Archive upload safety

X archive files can contain sensitive data beyond public posts. Treat uploads as high risk.

Requirements:

- Accept only expected archive files or a strict extracted subset.
- Parse allowlisted post data only.
- Never parse or display DMs.
- Never store full raw archives longer than needed.
- Delete raw archive uploads after parsing.
- Encrypt parsed post content if stored.
- Provide a purge all local archive data action.

## Deletion job safety

A real deletion job may start only when all conditions are true:

- User is authenticated.
- X token belongs to the account shown in the UI.
- Queue was generated by a server-side scan or archive parse.
- Dry run completed.
- Queue hash matches the reviewed queue.
- Account handle is shown in final confirmation.
- Post count is shown in final confirmation.
- Typed deletion verse matches the expected verse.
- Final literal line matches exact destructive action.
- Job is rate limited.
- Job is logged.

## Queue hash

When a burn bag is sealed, compute a hash of the ordered post IDs and filters.

Pseudo format:

```txt
queue_hash = sha256(account_id + sorted_post_ids + filters + created_at_bucket)
```

Store this server-side. Require the same hash at job start.

## Rate limits and batch behavior

- Delete in small batches.
- Respect X API rate limits and retry guidance.
- Use exponential backoff for transient failures.
- Stop on token revocation.
- Stop on repeated authorization errors.
- Provide pause and abort for remaining work.
- Do not retry indefinitely.

## Logging

Log:

- Job ID
- User ID
- X user ID
- Queue hash
- Filter summary
- Post IDs attempted
- Success or failure per post
- Error class
- Timestamp

Do not log:

- Access tokens
- Refresh tokens
- Raw archive files
- Full post text unless user explicitly exports a local report

## Product safety copy

Every final destructive screen must include:

```txt
This app cannot restore posts after X confirms deletion.
Pause and abort only stop remaining work.
Already deleted posts are gone from this app's control.
```
